# SIEM Detection Rules and Alert Investigation

## Overview
A SIEM detects threats by using detection rules. These rules examine incoming logs and trigger alerts when specific suspicious conditions are met.

A SIEM does not automatically know that an activity is malicious. It depends on predefined logic written as rules.

## Basic Flow
Logs are collected from different systems and sent to the SIEM.  
The SIEM checks those logs against detection rules.  
If a rule matches, the SIEM generates an alert.  
The analyst then investigates the alert.

**Flow:**  
`Logs -> Detection Rule -> Alert -> Investigation`

## What is a Detection Rule?
A detection rule is a logical condition created to identify suspicious behavior, policy violations, or signs of attack.

These rules monitor specific log fields and their values. When the rule conditions are satisfied, an alert is triggered.

## Examples of Detection Rules
- If a user has 5 failed login attempts in 10 seconds, trigger an alert for **Multiple Failed Login Attempts**
- If a login becomes successful after many failed login attempts, trigger an alert for **Successful Login After Multiple Failed Login Attempts**
- If a user plugs in a USB device, trigger an alert if USB use is restricted by company policy
- If outbound traffic is greater than 25 MB, trigger an alert for **Potential Data Exfiltration Attempt**

## How is a Detection Rule Created?
Detection rules are built using field-value pairs from logs.

A SIEM looks at important fields such as:
- log source
- event ID
- process name
- username
- IP address
- traffic volume

The rule is then written as a condition using those values.

## Detection Rule Use Case 1: Event Log Cleared
Attackers may clear logs after exploiting a system in order to hide evidence.

In Windows, **Event ID 104** is generated when event logs are cleared.

### Rule
If:
- `LogSource = WinEventLog`
- `EventID = 104`

Then:
- Trigger alert: **Event Log Cleared**

### Why it matters
This may indicate an attacker trying to remove traces of their activity.

## Detection Rule Use Case 2: WHOAMI Command Execution
Attackers often run commands like `whoami` after gaining access to a system or escalating privileges.

In Windows, **Event ID 4688** is commonly associated with process creation.

### Helpful fields
- `LogSource`
- `EventID`
- `NewProcessName`

### Rule
If:
- `LogSource = WinEventLog`
- `EventID = 4688`
- `NewProcessName contains whoami`

Then:
- Trigger alert: **WHOAMI Command Execution Detected**

### Why it matters
This may indicate post-exploitation activity or attacker reconnaissance on the compromised system.

## Why Normalized Logs Matter
Detection rules depend on specific fields being available and consistent.

If logs are not normalized properly, the SIEM may not correctly understand:
- event IDs
- usernames
- process names
- source IPs
- actions performed

### Conclusion
Normalized logs are important because detection rules rely on field-value pairs to work correctly.

## Alert Investigation
After an alert is triggered, the analyst investigates the related events and checks which rule conditions were met.

Analysts usually monitor alerts through SIEM dashboards, which provide a summarized view of important activity across the environment.

During investigation, the analyst examines:
- the alert details
- the related logs or flows
- the rule that triggered
- whether the activity is actually suspicious

## True Positive vs False Positive

### False Positive
A false positive means the alert was triggered, but the activity is actually harmless or legitimate.

### Action
- tune or adjust the rule
- reduce similar false alerts in the future

### True Positive
A true positive means the alert correctly identifies suspicious or malicious activity.

### Action
- perform deeper investigation
- contact the asset owner
- confirm whether the activity is authorized
- isolate the infected host if needed
- block the suspicious IP address

## Main Concept
A SIEM detects threats by applying detection rules to normalized logs. When suspicious conditions match a rule, an alert is generated. The analyst then investigates the alert and decides whether it is a true positive or a false positive.

## Final Summary
- SIEM alerts are generated by detection rules
- detection rules are logical expressions based on log fields and values
- normalized logs are necessary for rules to work correctly
- alerts must be investigated by analysts
- analysts decide whether an alert is a false positive or a true positive
- based on the investigation, action is taken to protect the environment
